### EXP-2026W29-001

- 源意图 (Original Vibe): 按 PI 批准的计划（`~/.claude/plans/design-another-round-of-ancient-island.md`）修三个瓶颈（吞吐/稳定/度量）后**有力检验 H1**：训 dense/sparse 再配对 OOD 判 per-step vs terminal。
- 假设 (Hypothesis): 稳定化 + 更大 batch 训出的 dense 在 **OOD ASR** 上 > sparse（且两臂 > base）。
- 是否被驳斥 (Falsified?): 未判（**基建 + 训练完成；判决 eval 因 GPU 被回收待恢复**）
- 驳斥/支持原因 (Why):
  - **基建（CPU 全验证）**：`rollout_batch`（逐轮 lockstep 批量生成 + 并发 victim）**≡ 顺序 rollout**（等价金标准过）；KL-to-ref 经 PEFT `disable_adapter`（无需第二模型）；配对分析复用 `analysis.bootstrap_diff` + 新增 bootstrap-p + Holm（合成数据验 SUPPORTED/null 判定正确）。
  - **吞吐达标**：throughput smoke **438s/64-rollouts ≈ 6.8s/rollout vs 旧 37.5s/rollout = 5.5×**，GPU 67.5GB 无 OOM。
  - **稳定化生效**：加 KL(0.01)+降 LR → **grad_norm ~20（旧 76）、无 collapse**（vs EXP-2026W28-009 dense 升后崩）。注：先跑 LR 3e-6 太保守（mean_phi 平在 0.3、KL~0.003 策略几乎不动）→ 改 LR 1e-5 重跑。
  - **训练中已见 H1 机制**：dense ~180 examples/step（信号丰）；**sparse 信号饥饿**——full-success（全 5 字段）仅 ~2% → 多数 terminal 奖励=0 → 优势多为 0 → 仅 ~24 examples/step（success 升到 12% 后回 ~110）。正是 per-step vs terminal 的差别。两臂 mean_phi 噪声大（~0.24–0.40）。
  - **未判原因**：训练完（`ALL_TRAIN_DONE`，双 adapter 落盘 H20 持久盘）后启 OOD+in-domain eval 时，**AutoDL 回收 GPU**（容器在、`nvidia-smi` No devices、vllm/victim 死）。eval 需 GPU（8B attacker + 27B victim）→ 待 GPU 恢复（用户告知几小时后回）。
- Agent 动作 (What changed): `mt_rollout.py::rollout_batch`（+ CPU 等价测试）；`h1_mt_grpo_train.py` 全批量化 + KL-to-ref + LR/steps/batch + step ckpt；`h1_mt_ood_eval.py` `--split/--seeds/best-of-K` + 批量;`h1_mt_powered_analyze.py`（bootstrap_diff + Holm 判决）；`h1_deploy_mt.py` 加 eval；`method.md §2` KL 项（PI 批）；`extraction_multifield.py` RNG 复现修复。adapter 备份本地（dense 完整；sparse 待 GPU 回稳定重拉）。
